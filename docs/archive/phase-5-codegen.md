# Phase 5: Code Generation Engine

## Goal

Implement MiniJinja-based template rendering. The engine loads templates from the domain directory and renders them with extracted module data. Create the initial firmware Rust templates.

## Engine setup

```rust
pub fn create_template_env(domain_dir: &Path, language: &str) -> Result<Environment, EngineError> {
    let template_dir = domain_dir.join("templates").join(language);
    let mut env = Environment::new();
    env.set_loader(path_loader(&template_dir));
    env.set_trim_blocks(true);
    env.set_lstrip_blocks(true);
    env.set_keep_trailing_newline(true);
    // No auto-escape — we're generating code, not HTML
    Ok(env)
}
```

## Standard filters

Registered on every template environment regardless of domain:

| Filter | Input | Output | Example |
|---|---|---|---|
| `snake_case` | `"AudioPipeline"` | `"audio_pipeline"` | `{{ name \| snake_case }}` |
| `pascal_case` | `"audio_pipeline"` | `"AudioPipeline"` | `{{ name \| pascal_case }}` |
| `screaming_snake` | `"AudioPipeline"` | `"AUDIO_PIPELINE"` | `{{ name \| screaming_snake }}` |
| `map_type` | `"Integer"` | `"i32"` (from type_map) | `{{ type \| map_type }}` |

`map_type` is parameterized by the domain's `type_map` config for the active language.

## Generation pipeline

```rust
pub fn generate(
    extraction: &ExtractionResult,
    config: &DomainConfig,
    language: &str,
    output_dir: &Path,
) -> Result<GenerationReport, EngineError>;
```

1. Create MiniJinja environment for the target language
2. Register standard filters + `map_type` with the domain's type map
3. For each extracted module:
   a. Compute spec hash (SHA256 of the serialized `ExtractedModule`)
   b. Check if output file exists with matching hash → skip if unchanged
   c. Render `module.<ext>.j2` with module data as context
   d. Write output file with hash comment header
4. For each extracted state machine:
   a. Same skip-if-unchanged logic
   b. Render `state_machine.<ext>.j2`
5. Return generation report

## Spec-hash fingerprinting

Each generated file includes a hash comment:

```rust
// Generated by sysml-v2-analyzer — DO NOT EDIT
// spec-hash: a1b2c3d4e5f6...
```

On subsequent runs, the engine reads the hash from existing files and skips regeneration if the input data hasn't changed. This makes `generate` fast and safe for incremental workflows.

## Generation report

```rust
pub struct GenerationReport {
    pub generated: Vec<GeneratedFile>,
    pub skipped: Vec<SkippedFile>,
    pub errors: Vec<GenerationError>,
}

pub struct GeneratedFile {
    pub template: String,
    pub output_path: PathBuf,
    pub module_name: String,
}

pub struct SkippedFile {
    pub output_path: PathBuf,
    pub reason: SkipReason,  // Unchanged | AlreadyExists
}
```

## Firmware Rust templates

Initial templates for the firmware domain (`domains/firmware/templates/rust/`):

### `module.rs.j2`

Renders a module with:
- Module-level doc comment from metadata
- Struct definition with config fields
- Constructor (`new`) with hardware init
- Interface methods (from actions)
- Error enum

### `state_machine.rs.j2`

Renders a state machine with:
- State enum with variants
- Event enum with variants
- Transition function (`fn handle_event(&mut self, event: Event) -> Result<()>`)
- Match arms for each (state, event) → next_state

### `test.rs.j2`

Renders test stubs:
- One test per interface method
- One test per state transition
- Setup/teardown helpers

## Template context

Each template receives a `minijinja::Value` containing:

```json
{
  "module": { /* ExtractedModule fields */ },
  "domain": { /* DomainConfig fields */ },
  "language": "rust",
  "timestamp": "2026-03-20T..."
}
```

Templates access data as `{{ module.name }}`, `{{ module.metadata.MemoryModel.allocation }}`, etc.
